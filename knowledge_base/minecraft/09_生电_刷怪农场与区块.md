# 生电：刷怪农场、区块加载与性能

标签：生电、刷怪塔、铁傀儡农场、区块加载、模拟距离、实体、TPS、Java 版、基岩版

## 刷怪农场的前提

刷怪效率不是只由“刷怪平台面积”决定。它同时受玩家位置、可生成空间、生物上限、亮度、区块加载、实体清理和版本规则影响。设计前先确定目标版本与农场类别：敌对生物、史莱姆、铁傀儡、村民作物、末影人或下界生物的条件不同。

## Java 版敌对生物的距离概念

常见 Java 版设计会把玩家停留点与刷怪平台保持在大约 24 到 128 格的有效区间内：距离过近通常不满足普通敌对生物生成条件，距离过远则容易不再稳定参与生成或被清理。具体球形距离、垂直距离和模拟距离会因版本、服务器设置及生物类别而变化，因此应把它作为设计起点而不是不可变公式。

```text
farm_rate ≈ eligible_spawn_spaces
          * spawn_attempt_success_rate
          * mob_cap_availability
          * collection_efficiency
```

任何一项接近零，农场产量都会明显下降。洞穴未照明、附近地表、其他玩家所在维度和服务器实体限制都会降低 `mob_cap_availability`。

## 区块、加载与模拟距离

区块被加载不代表所有机制都在完整模拟。服务器通常分别控制视距、模拟距离、实体激活和区块票据。生电装置应明确：玩家是否必须在场、是否需要加载器、离线后是否继续运行、以及服务器是否允许加载器。

```text
if chunk_is_loaded and within_simulation_distance:
    run_redstone_and_block_ticks()
else:
    freeze_or_reduce_mechanics()
```

加载器的实现与稳定性高度版本相关，也可能被服务器规则禁止。不要假设单人世界有效的加载器可直接用于公共服务器。

## 铁傀儡农场

铁傀儡农场通常依赖村民、床、工作站和威胁/睡眠等条件的组合，但 Java 版与基岩版的核心条件不同。选择设计时必须使用目标版本专用教程。调试顺序是：确认村民可寻路或可识别床/工作站、确认生成区域合法、确认铁傀儡没有卡在附近未被收集、确认玩家和区块保持模拟。

## 性能预算

生电服务器的性能通常消耗在实体、漏斗、方块实体、红石更新和高频命令上。可以用下列顺序排查：

```text
1. 大量掉落物、经验球、矿车和生物实体
2. 长漏斗线与持续扫描的容器
3. 高频时钟、观察者循环和活塞飞行器
4. 未限制的刷怪区与多玩家并行农场
5. 命令方块或数据包中的大范围实体选择器
```

高效不等于一直运行。为大型农场加总开关、实体上限、收集/销毁分流和维护模式，通常比单纯增加平台规模更可靠。
