# 生电：测试方法、版本差异与故障定位

标签：生电、测试世界、Java 版、基岩版、准连接、更新顺序、TPS、排错

## 先建立测试世界

复杂装置不要直接在主存档或公共服务器搭建。测试世界至少应包含：与生产环境相同的版本、相同的游戏规则、可查看红石强度的调试线路、可切换的加载距离，以及可复制的原型区域。

## 装置状态机写法

把装置先写成状态与转移，再搭建方块。以下模板适用于收割机、门控刷怪塔和分类系统：

```text
STATE_IDLE:
    wait for trigger
    -> STATE_ACTIVE

STATE_ACTIVE:
    enable collection or movement
    wait for completion signal or timeout
    -> STATE_COOLDOWN

STATE_COOLDOWN:
    reset pistons, hoppers and locks
    wait for safe delay
    -> STATE_IDLE
```

每个状态应有可观察输出，例如指示灯、比较器读数或测试箱库存。这样能够区分“触发条件没到”“电路没更新”“收集堵塞”和“延迟不够”。

## Java 版特有风险

Java 版复杂红石常会涉及准连接、方块更新顺序、活塞顺序和实体碰撞。它们会让同一逻辑图在旋转、镜像或区块边界处表现不同。不要把依赖未明确更新顺序的设计当作长期稳定基础设施；关键装置应选择版本标注清楚、容错更高的方案。

## 基岩版特有风险

基岩版红石更新、活塞行为、刷怪与村民/铁傀儡条件和 Java 版存在显著差异。看到“Java only”“Bedrock only”标识时，应直接寻找对应版本方案，而不是通过少量改线强行移植。

## 统一故障定位表

| 症状 | 优先检查 | 常见原因 |
| --- | --- | --- |
| 装置完全不动 | 电源、朝向、区块模拟 | 中继器反向、比较器模式错误、区块卸载 |
| 偶尔失效 | 时序、脉冲长度、实体拥堵 | 脉冲过短、服务器低 TPS、漏斗锁定 |
| 产量很低 | 玩家位置、附近刷怪、实体上限 | 未清理洞穴、模拟距离不足、其他玩家分流 |
| 分类串线 | 过滤库存、溢流、物品堆叠 | 占位物错误、漏斗满载、版本物品数据差异 |
| 服务器卡顿 | 实体、漏斗、循环红石 | 未限速农场、高频时钟、大范围扫描 |

## 发布前检查清单

```text
[ ] 在目标版本测试过
[ ] 断开玩家或卸载区块后的行为已确认
[ ] 容器满载和溢流时不会反向堵塞
[ ] TPS 下降时不会产生无限循环或物品喷射
[ ] 有手动总开关和维护入口
[ ] 有备份或原理图，便于重建
```
